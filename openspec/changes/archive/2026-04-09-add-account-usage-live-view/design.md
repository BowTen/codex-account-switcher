## Context

`codex-auth usage` 现在的批量查询已经有并发能力，但 CLI 仍然等到所有账号都完成后再一次性打印，这让命令在真实网络环境里看起来像“没有输出、像是卡住了”。用户已经明确要求把 bare `usage` 提升为一个实时终端视图：查询状态要可见、完成结果要边查边出现、已完成账号还要按额度压力动态排序。

这个 change 只改 bare `codex-auth usage` 的终端交互。named usage 查询、usage API 合约、token refresh 和账号持久化语义保持原样，避免把展示层变更扩散到认证链路。

## Goals / Non-Goals

**Goals:**
- 在 TTY 中为 bare `codex-auth usage` 提供实时重绘视图。
- 在底部显示稳定的状态区，持续展示当前 phase、running 账号和 queued 账号。
- 已完成账号立即插入结果区，而不是等整批结束。
- 把成功结果按 `5h remaining`、`weekly remaining` 升序排列，让额度最紧张的账号靠上。
- 保留非 TTY 的纯文本回退路径，避免破坏重定向和脚本使用。

**Non-Goals:**
- 不把 named `codex-auth usage <name>` 改造成全屏交互视图。
- 不新增 curses 或第三方 TUI 依赖。
- 不新增排序、并发或显示风格参数。
- 不改变 usage payload、本地快照格式或 refresh 协议。

## Decisions

### 1. 只在 `stdout` 是 TTY 且运行 bare `usage` 时启用实时视图

实时重绘对交互终端有价值，但对管道、重定向和脚本消费来说会制造噪音。因此 live view 只在 `sys.stdout.isatty()` 为真且运行 bare `codex-auth usage` 时启用，其余场景继续走一次性文本输出。

备选方案：
- 所有 bare `usage` 都强制使用实时视图。

拒绝原因：
- 会破坏 `codex-auth usage > file.txt` 和 `codex-auth usage | sed ...` 这类已有使用方式。

### 2. 让 service 发出进度事件，CLI 只负责渲染

实时视图需要知道哪些账号 queued、哪些 running、哪个刚完成，还需要知道 batch 是否被 timeout 中止。最稳妥的做法是让 service 层暴露事件流，把并发和状态推进保留在原来的执行层；CLI 只消费事件并重绘屏幕。

备选方案：
- 让 CLI 自己并发调 service 的单账号接口。

拒绝原因：
- CLI 会被迫管理线程池、running 队列和中止逻辑，和现有职责边界冲突。

### 3. 未完成账号始终停留在底部状态区，不在结果区占位

用户已经明确要求“查询状态也一直位于下方”。因此未完成账号只在状态区出现，完成后再进入结果区。这样动态排序只影响已完成结果，不会让未完成账号的占位行在屏幕中来回跳动。

备选方案：
- 所有账号先占位显示在结果区，完成后原地更新并重排。

拒绝原因：
- 屏幕抖动更大，也和“查询状态一直位于下方”的要求相反。

### 4. 错误结果置顶，成功结果再按额度压力排序

成功结果排序规则固定为：
- `5h remaining` 升序
- `weekly remaining` 升序
- 账号名稳定打平

但错误账号优先级更高，应始终位于成功结果之前，避免它们被“额度排序”埋掉。

备选方案：
- 让错误账号也参与同一套额度排序。

拒绝原因：
- 错误账号通常没有完整额度窗口，不适合和成功结果混排。

## Risks / Trade-offs

- [实时重绘会增加 CLI 复杂度] → 把事件生产和渲染分层，继续保留非 TTY 回退路径。
- [动态排序会让已完成结果在屏幕中移动] → 只对已完成结果排序，未完成账号固定留在底部状态区，减少视觉抖动。
- [ANSI 重绘序列在某些环境可能表现不一致] → 限制 live view 只在 TTY 启用，并确保重绘失败时可退回普通文本输出。
